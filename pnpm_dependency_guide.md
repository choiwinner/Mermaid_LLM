# 🍱 pnpm의 의존성 관리 방식과 핵심 파일의 이해

`pnpm`으로 프로젝트를 관리할 때 필수적으로 마주하게 되는 세 가지 파일(`package.json`, `pnpm-workspace.yaml`, `pnpm-lock.yaml`)의 역할과 관계를 이해하기 쉽게 정리한 가이드입니다.

---

## 👨‍🍳 요리에 비유한 핵심 파일의 역할

프로젝트를 하나의 **'요리'**라고 생각하면 이 파일들의 관계가 명확해집니다.

| 파일명 | 요리 비유 | 실제 역할 (Technical Role) |
| :--- | :--- | :--- |
| **`package.json`** | **장바구니 메모지** | **의도(Intent):** 개발자가 어떤 라이브러리가 필요한지 명시한 목록입니다. |
| **`pnpm-workspace.yaml`** | **주방 설계도** | **범위(Scope):** 모노레포 구조에서 내부 패키지들의 경로를 정의합니다. |
| **`pnpm-lock.yaml`** | **구매 영수증** | **결과(Snapshot):** 설치된 모든 의존성의 정확한 버전과 관계를 기록합니다. |

---

## 1. package.json (무엇을 살 것인가?)
*   **사람(개발자) 중심:** "나는 lodash 4버전 대가 필요해"라고 내 의도를 적어두는 곳입니다.
*   **필요성:** 이 파일이 없으면 `pnpm`은 당신이 어떤 도구를 쓰려 하는지 알 수 없습니다.
*   **특징:** 버전 범위(`^4.17.0`)를 적기 때문에, 시간이 흘러도 이 파일 내용만으로는 매번 똑같은 설치를 보장할 수 없습니다.

## 2. pnpm-workspace.yaml (우리 집 주방은 어디까지인가?)
*   **구조 중심:** "내 프로젝트는 `packages/` 폴더 안의 내용도 포함해"라고 구역을 정합니다.
*   **필요성:** 이 프로젝트처럼 **모노레포(Monorepo)**인 경우, 내부 패키지끼리 서로를 참조할 수 있도록 연결해 주는 '지도' 역할을 합니다.
*   **특징:** 외부 라이브러리가 아닌 "우리가 직접 만든 코드"를 관리할 때 필수적입니다.

## 3. pnpm-lock.yaml (실제로 무엇을 사 왔는가?)
*   **기계(pnpm) 중심:** `package.json`을 보고 실제로 설치를 완료한 뒤, 그 **정확한 시점의 결과**를 캡처해둔 파일입니다.
*   **필요성:** "서울우유 900ml 저지방 버전, 구매 일시 2026-03-29"처럼 아주 상세한 '영수증'이므로, 전 세계 어떤 개발자가 설치해도 **100% 동일한 환경**을 만들어줍니다.
*   **특징:** 사람이 직접 수정하지 않으며, 보안 및 환경 일관성(Reproducibility)의 핵심입니다.

---

## 🔄 설치 과정의 흐름

1.  **`pnpm install` 실행**
2.  `pnpm-workspace.yaml`을 보고 프로젝트의 **전체 지도**를 그립니다.
3.  `package.json`에서 개발자의 **요청 목록**을 확인합니다.
4.  `pnpm-lock.yaml`을 확인하여 예전에 기록된 **정확한 버전**이 있는지 봅니다.
5.  기록이 있다면 그대로 설치하고, 기록이 없거나 내용이 바뀌었다면 새로 계산하여 **영수증(lock 파일)을 갱신**합니다.
